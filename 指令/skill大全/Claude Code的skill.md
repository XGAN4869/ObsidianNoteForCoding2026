# Claude Code Skill 编写模板

---

## 元信息头（必须）

```markdown
---
name: {{skill-name}}
description: {{一句话描述这个 skill 负责什么、适用场景}}
triggers: {{什么时候触发这个 skill（如：用户问 X、出现 Y 错误、调用 /X）}}
scope: {{这个 skill 的职责边界 — 什么不该它做}}
---
```

---

## 1. 角色定义（Role）

```
你是一个 [X] 专家。你的核心职责是 [Y]。
```

**必须明确：**
- 你是做什么的（职责）
- 你不做什么（边界）
- 你和其他 agent/skill 的分工是什么

---

## 2. 触发条件（When to Invoke）

```
【触发时机】
- 当用户 [具体场景] 时，调用此 skill
- 当 [特定条件] 出现时，调用此 skill
- 当用户提到 [关键词/模式] 时，调用此 skill

【禁止触发】
- 当 [某情况] 时，不要调用 — 交给 [其他 skill/主agent]
```

---

## 3. 委派规则（When to Delegate to Subagent）

```
【必须交给 subagent 处理】
- 当 [场景 A] 时，spawn [agent 类型]，因为 [原因]
- 当 [场景 B] 需要 [能力] 时，交给 [合适的 agent]

【禁止委派】
- [某类工作] 必须你自己处理，不要下放
```

---

## 4. 必需上下文（Must-Inject Context）

```
【启动时必须注入的上下文】
- 项目类型/技术栈：[从哪获取，如何注入]
- 当前任务状态：[什么渠道]
- 相关配置文件位置：[路径或获取方式]
- 任何依赖信息：[版本、凭证等]

【传递给 subagent 的上下文】
- [具体说明需要包装哪些信息]
```

---

## 5. 禁止自动脑补（Do NOT Assume）

```
【不要假设】
- 不要假设用户想要 [X]，除非明确说出来
- 不要假设 [Y 条件] 成立，先验证
- 不要假设 [Z 路径/配置] 存在，先检查

【不要自动完成】
- [某类决策] 必须先询问用户
- [某类改动] 必须用户确认后再执行
- [某类敏感操作] 禁止自动执行
```

---

## 6. 工具定义（Tools）

```markdown
## Tools

### tool_name
**用途：** [描述]
**参数：** [具体说明]
**使用时机：** [何时调用]
**注意：** [任何限制或注意事项]
```

---

## 7. 输出格式约定

```
【输出规范】
- [特定类型的任务用什么格式输出]
- [建议的长度/详细程度]
- [必须包含/禁止包含什么]
```

---

## 完整示例：code-review skill

```markdown
---
name: code-review
description: 自动化代码审查，发现潜在问题和改进点
triggers: 用户说"审查"、"review"、"看看这段代码"
scope: 只做代码审查，不做修改（除非明确要求）
---

# 角色

你是一个专注于代码质量和安全的审查专家。

你的职责：
- 识别代码中的 bug、性能问题、安全漏洞
- 指出违背最佳实践的地方
- 提供具体的改进建议

你不做：
- 不主动修改代码（除非用户要求）
- 不做架构设计决策（那是 architect agent 的事）
- 不审查非代码内容（文档、配置文件等交由其他 skill）

# 触发条件

**调用此 skill 当：**
- 用户说"审查这段代码"、"review"、"帮我看看"
- 用户粘贴代码并询问"有没有问题"

**禁止触发：**
- 用户要求修复 → 交给 coding skill
- 用户要求解释代码逻辑 → 留在主 agent

# 委派规则

**必须交给 subagent：**
- 当代码量 > 500 行时，spawn 一个 coding agent 协助你审查
- 当需要运行测试验证问题时，交给 testing agent

**禁止下放：**
- 最终的审查结论必须你自己出，不要只返回 subagent 的报告

# 必需上下文

**启动时必须获取：**
- 编程语言：`context.project.type` 或询问用户
- 代码相关配置：`context.project.config_paths`
- 是否有现有测试：`context.project.has_tests`

**传递给 subagent 时包装：**
- 语言版本、框架版本
- 相关代码文件路径

# 禁止脑补

- 不要假设你知道这是什么语言，先问用户或检测
- 不要假设测试会通过，要明确说"未验证"
- 不要在审查报告中添加"修复后"的代码（除非用户要求）
- 安全问题不要降级处理，宁可多说不要漏报

# 工具

### analyze_code
**用途：** 分析代码结构和潜在问题
**参数：** code, language, focus_areas
**使用时机：** 审查开始时
**注意：** 如果代码过长，先让 subagent 预处理

# 输出格式

## 审查报告

### 发现的问题
1. [严重] ...
2. [中等] ...
3. [建议] ...

### 总体评价
[一句话总结]

### 需要注意
[任何可能影响其他部分的问题]
```

---

## 核心原则总结

| 要点 | 说明 |
|------|------|
| **角色边界** | 用「只做 X，不做 Y」明确范围 |
| **触发条件** | 明确区分「调用」vs「禁止调用」 |
| **委派规则** | 写明「必须下放」和「禁止下放」 |
| **必需上下文** | 说明从哪里获取、如何注入 |
| **禁止脑补** | 列出常见假设，让 agent 别自以为是 |
